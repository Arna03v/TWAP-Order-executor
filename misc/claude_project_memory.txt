Order Executor take-home task: Optimise for the rubric — (1) correct async TWAP with accurate partial-fill progress accounting, (2) sound progress/result interface design with justification, (3) clean exchange adapter separating logic from I/O, (4) sensible shutdown behavior, (5) tests, code clarity, good scope judgment. 2. Skip optional frontend/web UI entirely. Progress display will be terminal-only (caller prints progress to stdout).
3. Order Executor — Language: Go. Reasons: (a) goroutines + channels are the most natural model for background TWAP with streamed progress, (b) context.Context is purpose-built for cancellation/shutdown, (c) interfaces are first-class for the exchange adapter, (d) built-in testing with table-driven idiom, (e) user knows Go well and can walk through every line in a live interview. README must explain Go idioms briefly for potentially Python-native evaluators.
4. Order Executor — README must cover these explanations: (1) How to run (exchange + executor + caller), (2) Progress/result interface design and WHY that approach over alternatives, (3) How failure is reported, (4) Exchange adapter — what the interface looks like and how it enables testability, (5) Clean shutdown choice and WHY, (6) What you'd do next if more time. These map directly to the 5 evaluation criteria in the rubric.
5. Order Executor — Concurrency decisions: (1) In-process library, not a service. Caller imports executor package and calls functions directly. No network layer between caller and executor. (2) Single coordinator goroutine with two tickers: placeTicker (duration/slices interval) to POST slices, pollTicker (~200ms) to GET active order statuses. One select loop, no child goroutines, no shared state, no mutex, no WaitGroup. (3) context.Context + cancel func for shutdown. Single mechanism. Coordinator select includes ctx.Done case — cancels active orders on exchange, computes final state, exits. (4) Child failure: retry (configurable count via env/variable), coordinator tracks retry count per slice. If all retries fail, log failure, drop that slice, continue. Report shortfall in final result. (5) Exit condition: placeTicker stops after all slices posted. pollTicker runs until all active orders resolved (filled/failed/cancelled). Coordinator sends final result and returns. (6) Caller scope: single TWAP order, print progress, print final result, exit. Executor supports concurrent orders by design but caller doesn't exercise it.
6. Order Executor — "What I'd do next" items for README: (1) Spawn HTTP calls (POST/GET to exchange) as goroutines so blocking I/O doesn't delay tickers in the select loop. Not needed for localhost mock but matters at real network latencies.
7. Order Executor — Extensibility via Strategy interface: Strategy interface has Execute(ctx, exchange) <-chan ProgressUpdate. Three extensibility axes: (1) New strategy types (VWAP, Iceberg) = new struct implementing Strategy with its own Execute. No changes to caller or progress channel. (2) Child order type within a strategy (market vs limit TWAP) = OrderType field on TWAPStrategy struct, branching inside Execute. (3) Progress channel is strategy-agnostic — caller reads same <-chan ProgressUpdate regardless of strategy. Caller creates strategy directly, no Executor wrapper for this assignment. "What I'd do next": an Executor manager struct that tracks multiple concurrent strategy runs with unified shutdown.
8. Order Executor — Progress/result interface design: (1) Single channel <-chan ProgressUpdate returned by Execute. Caller reads with for-range, loop exits on channel close. Last update is the final result. (2) Justification over alternatives: polling requires caller-side loop + mutex on shared state; callbacks couple caller code into coordinator goroutine; channel decouples producer/consumer, no mutex needed, channel IS the synchronization. (3) Channel is push-based streaming — naturally hits the stretch goal "richer progress channel / streaming instead of polling." Call this out in README. (4) Dedup: hash last-sent state, hash current state, only send on channel if different. Avoids flooding caller with identical updates. (5) ProgressUpdate fields: TotalQty, FilledQty, AvgPrice, SlicesTotal, SlicesPlaced, SlicesFilled, Status (running/success/failed/cancelled/partial), Errors []string (final per-slice failure descriptions), Logs []string (delta — informational events in this update only, e.g. retry attempts; empty on most updates), Timestamp. (6) Status "partial" = some slices failed but rest succeeded. (7) StatusCompleted renamed to StatusSuccess.
9. Order Executor — Test cases to cover: (1) Aggregate weighted avg price calculation: sum(child_filled_qty × child_avg_price) / total_filled_qty — easy to get wrong, must have a test. (2) Partial fills — child order fills in multiple chunks, progress updates reflect intermediate state. (3) Failed child order — retry logic, fallthrough to failure, shortfall in final result. (4) Clean shutdown mid-execution — context cancelled, active orders cancelled on exchange, final update has correct filled qty. (5) All slices fill successfully — happy path. (6) Slicing math — total_qty / slices produces correct per-slice qty including remainder handling.
10. Order Executor — Exchange adapter design: (1) Exchange interface with 3 methods: PlaceOrder(ctx, req) (*Order, error), GetOrder(ctx, orderID) (*Order, error), CancelOrder(ctx, orderID) error. GetBBO dropped — not used by market-order TWAP, can be added back when limit orders are implemented. (2) Two boundaries in the system: Caller ←function calls→ Executor ←HTTP calls→ Mock Exchange. Adapter wraps the second boundary. Caller creates the adapter, passes it to strategy's Execute(ctx, exchange). (3) Two implementations: HTTPExchange (real HTTP to mock exchange, used by caller) and FakeExchange (scripted deterministic responses for tests). (4) Why not test against real mock: non-deterministic fills, timing-dependent, flaky. FakeExchange gives instant deterministic tests. (5) Adapter is internal to executor package. Caller only sees Strategy interface. (6) PlaceOrderRequest and Order have JSON tags matching mock server field names (snake_case). No internal wire types — domain types marshal/unmarshal directly. ClientOrderId has omitempty so empty string is not sent to server.
11. Order Executor — TWAP internals and shutdown details: (1) Remainder handling: total_qty / slices, last slice gets the remainder (e.g. 10/3 = 3.33, 3.33, 3.34). (2) client_order_id: use exchange's idempotency key for safe retries — if POST succeeds but response lost, re-POST with same client_order_id returns existing order instead of duplicate. PlaceOrderRequest has optional ClientOrderID field, any strategy can use it. (3) Timeout context on HTTP calls: prevent hung network call from blocking the coordinator select loop. (4) Shutdown sequence: stop placing new slices → DELETE all open/partially-filled orders on exchange (prevents orphaned orders) → compute final state from what actually filled → send final ProgressUpdate with status cancelled → close channel → exit. Best-effort DELETE with short timeout — don't block shutdown if exchange is unresponsive. (5) Why DELETE: orphaned orders with no observer keep filling and cause unexpected positions. Cancelling is the safe default.

13. Order Executor — Error handling decisions: (1) GetOrder fails during polling: skip that order for this poll tick, retry on next tick (~200ms). Do NOT count as a failure or increment retry counter — the order still exists on exchange, just couldn't read status. Optionally log it. (2) PlaceOrder fails: retry up to N times (configurable) with same client_order_id. If all retries fail, drop slice, continue with remaining. Add failure description to Errors slice. Final status is partial if some slices succeeded. (3) CancelOrder fails (shutdown): best-effort, proceed with shutdown anyway. Report last-known state in final ProgressUpdate. (4) General: return errors, never panic. Wrap with fmt.Errorf("...: %w", err).
